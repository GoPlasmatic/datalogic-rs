A visual language for JSONLogic

Read logic like a circuit, not a colour chart.

A rule is signal flowing through gates to a decision. This system colours nodes by the value that flows out of them — boolean, number, string, collection, data — so you can trace why a rule resolves the way it does. Flip the inputs and watch the board light up.

Or scrub the step debugger under the board — play, step, and rewind the evaluation; each stop parks a probe on the active node showing its context and result.

The problem with today's view

Eleven saturated hues, one hidden meaning

Today every operator category owns a colour and a full-width coloured header. With 15+ nodes the canvas becomes a rainbow where nothing recedes — and three of the eleven hues are near-identical purples, three more are near-identical cyans. Colour signals which family an operator belongs to, which is rarely the question you're asking. The question is almost always: what value does this produce, and is my condition true?

Reading nested logic

Follow the data, not the JSON

A deeply nested rule is hard to read when the diagram mirrors the JSON declaration: the root operator sits on the outside and you descend inward — the opposite of how the value is computed. The data-flow view inverts it. Sources enter on the left, the result leaves on the right, and every wire is coloured by the type it carries and tagged with its value. Sub-expressions that are just scalar maths collapse into one formula so the real joints stand out. This is Order Total from the sample set: reduce the cart, then apply a discount.

The one colour axis

Colour by the signal on the wire

Seven signal types instead of eleven operator families. Each maps to something you actually reason about while debugging: the type of value flowing out of a node. The palette is derived from GitHub Primer's syntax colours, proven legible on light and dark.

Operator category doesn't disappear — it moves to a monochrome glyph and a small-caps label, so power users keep the taxonomy without paying for it in colour.

Never colour alone. True carries a ✓, false an ✕; taken wires are solid-and-flowing, skipped ones dashed. Red/green truth survives deuteranopia because the shape inside the LED, not just its hue, tells you the answer.

Six rules the system follows

Shape encodes role. Colour encodes type.

01

Role is a silhouette

A boolean gate carries a signal LED and reads as infix math. A junction (and/or) is a compact connective with input ports. A branch has When / Then / Else rails. A value shows a typed output. You know what a node does before you read a word.

02

Operators are math glyphs

≥ ≤ ≠ ∧ ∨ ¬ ∈ × render at display size; the verbose label moves to a small-caps eyebrow. age ≥ 18 beats a stack of cards labelled "Greater Than Or Equal."

03

The board stays quiet

Chips are near-neutral with a 3px signal rail. Saturation is spent on the two things that matter: the type pill on an operand and the live state when data runs. Everything else recedes.

04

Truth is visible at rest

You don't need debug mode to see structure. When data is present, LEDs light true/false, taken wires carry animated current, and short-circuited branches visibly dim — the evaluation model becomes the diagram.

05

Slots are first-class

Every argument is a slot. Empty slots are dashed drop targets with a type hint. Leaves edit inline; the operator glyph is a click-to-swap target. Viewing and editing are the same surface at two zoom levels.

06

Both themes, one identity

Dark is a lit board; light is a schematic on paper. Same signal semantics, re-tuned for contrast — not a mechanical invert.

The shape vocabulary

A silhouette for every role — not one card for all

When every operator is a rectangle, shape carries no meaning and you read every node from scratch. Here each family has its own outline, borrowed from the notations engineers already know: logic-gate symbols for boolean combine, a flowchart diamond for branching, a round node for arithmetic, a stacked frame for iteration, a plug for data. You recognise the role from across the canvas, before reading a single glyph.

Reading a run, step by step

The debugger speaks the same language

Evaluation isn't a separate mode with its own palette. A transport scrubs the trace; each stop parks a probe on the active node; a side panel holds the raw logic, data and result. Every value keeps its signal colour from the board, to the probe, to the JSON.

Built for editing, not just viewing

The reading view is the editing view

Nothing switches into a separate "edit mode." Every visible token is a target: swap an operator, retype a path, fill an empty slot, or add an argument — all in place.

How it lands in the codebase

An evolution of the React Flow renderer, not a rewrite

The graph engine (@xyflow/react + dagre LR), the node/cell data model, the debug trace and the step-through controls all stay. What changes is the token layer and how a node decides its colour. Concretely:

AreaTodayProposed
colour source CATEGORY_COLORS[data.category] — 11 hues drive header + border + tint Add data.returnType → SIGNAL_COLORS. Category collapses to a monochrome glyph-badge + eyebrow.cat label.
node header Solid saturated .vertical-cell-header, full-width, white text, long label Near-neutral chip + 3px left signal rail (.node::before). Operator shown as a display math glyph; label demoted to eyebrow.
comparisons Card titled "Greater Than Or Equal" with stacked var/literal cells Infix gate: age ≥ 18 with a signal LED. Uses the existing ui.shortLabel, upgraded to math glyphs.
truthiness Only in debug: edge colour + amber pulse + result badge LED + border state on every boolean gate; short-circuited nodes dim (data-state="skipped"). Same trace data.
two colour tables constants/colors.ts and config/categories.ts duplicated Unify. Category keeps a glyph + name only; a single SIGNAL_COLORS owns the palette.
theme tokens styles/theme.css scoped to .logic-editor Same mechanism. Replace category vars with --sig-* signal vars; retune neutrals to the board/schematic pair.
edit affordances Properties panel + hidden handles Inline slots, glyph-as-menu, tap autocomplete — layered on the existing node without removing the panel.
return type Not surfaced — all/some/none look like collections; datetime like a number Each operator declares a returnType already in operators.types.ts. Drive the rail from it, add --sig-temporal for the 8th; quantifiers gain a truth LED.
node shape UnifiedOperatorNode — one card for every operator; nodeType unused for silhouette Map the existing ui.nodeType (decision, iterator, vertical, structure…) to a silhouette: case-ladder, quantifier frame, brace manifold, guard/octagon.
debugger UI DebuggerControls, DebugInfoBubble, DebugPanel on a separate token set (--debug-*, --accent-amber) Re-skin over the same JSX onto --sig-*: transport with a scrubber counter, node-anchored probe, signal-railed panel. A value is one colour board→probe→JSON.
error handling Only idle/true/false/skipped states; errors live in a panel Add data-state="error" (red halo + ⚠ badge + faulted wire), a try guard and a throw octagon. Error stays in the red family — told apart from false by form.

The one real bet: spend the entire colour budget on the value flowing through the wire, and demote operator category to monochrome. Node-graph editors (n8n, Node-RED, Retool) all colour by category; colouring by signal type is the opinionated departure, and it's the version that answers the question users actually ask while debugging a rule.
